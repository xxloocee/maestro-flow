# Maestro Flow MVP 定义

## 1. 文档目的

本文档用于定义 Maestro Flow 当前阶段的 MVP（最小可行产品）边界，面向对外发布的开源 CLI 工具场景。

本文档基于当前已实现能力整理，主要参考以下文档：

- `docs/P0_P1_implementation.md`
- `docs/P3_rollback.md`
- `docs/P4_知识库_提示词版本化_策略门禁.md`
- `docs/P5_策略插件化_与PR策略明细.md`
- `docs/P6_执行闭环_自动改代码_自动测试_自动修复.md`

本文档的目标不是重复描述所有实现细节，而是回答下面几个问题：

- 当前版本对外到底是什么产品
- 当前版本默认要解决什么问题
- 哪些能力属于 MVP 承诺
- 哪些能力虽然已经实现，但暂时不作为默认主路径

## 2. 产品定义

Maestro Flow 是一个独立的开源 CLI 工具，用于编排结构化的多 Agent 软件交付流程。

它的核心流程是：

1. 接收需求或 Spec 作为输入
2. 按固定的多 Agent DAG 执行研发流程
3. 产出结构化阶段结果和可追溯运行产物
4. 执行策略门禁与质量门禁
5. 为人工 Review 与最终交付决策提供依据

当前 MVP 的核心价值不是“全自动写代码并自动合并”，而是：

`把 AI 参与的软件交付过程变成可运行、可审查、可追溯的本地 CLI 工作流`

## 3. 目标用户

当前 MVP 面向的用户是：

- 希望在本地仓库中使用 CLI，而不是托管平台的开发者
- 希望把 AI 输出变成结构化产物，而不是只看聊天结果的开发者
- 需要保留运行记录、策略判断和 CI 结果的团队
- 愿意自己做最终代码审核和合并决策的用户

换句话说，MVP 面向的是“希望增强交付流程”的开发者，而不是“希望完全托管研发过程”的用户。

## 4. MVP 范围

### 4.1 属于 MVP 承诺的能力

以下能力属于当前版本对外可以明确承诺的范围：

- 基于 CLI 的需求驱动与 Spec 驱动运行入口
  - `run`
  - `spec init`
  - `spec run`
- 固定的多 Agent DAG 编排流程
  - `pm -> architect -> dev -> tester/debugger -> reviewer`
- 阶段依赖、有限并行、重试机制、状态机和错误码
  - 对应 P0/P1
- 可追溯的运行产物目录
  - `.maestro/runs/<run_id>/`
  - `run_state.json`
  - `stage_<name>.json`
  - `summary.md`
- 失败后的可配置回滚能力
  - 对应 P3
- 知识库注入、提示词版本化、策略门禁
  - 对应 P4
- 策略插件扩展、规则覆盖、PR/CI 策略结果展示
  - 对应 P5
- 基于运行产物的 CI 评估与 PR 评论生成
  - 当前代码中已实现，和既有规划方向一致

### 4.2 已实现但不作为默认主路径的能力

以下能力已经存在，并且有明确价值，但当前不作为 MVP 对外演示的默认路径：

- 自动应用 `dev.file_changes`
- 自动执行验证命令与测试命令
- Debugger 驱动的自动修复闭环
- `copy` 模式隔离执行
- 执行成功后的自动回写主工作区
- 人工决策式 sync-back 冲突处理

这些能力保留为进阶能力，不作为第一次上手体验的核心卖点。

## 5. 默认用户路径

当前 MVP 推荐的新用户路径应当是：

1. 安装依赖并配置模型提供方凭证
2. 先运行一次 mock 流程，确认环境可用
3. 再运行一次真实 requirement 或 spec 流程
4. 查看 `summary.md` 与 `run_state.json`
5. 基于 reviewer、policy、CI 结果进行人工交付判断

默认路径强调的是：

- 流程清晰
- 结果可追踪
- 产物可审查
- 决策有人控

默认路径不应该要求用户必须开启执行闭环，才能体现产品价值。

## 6. MVP 非目标

当前 MVP 明确不承诺以下能力：

- 完全自治的软件交付
- 无人工审核的自动合并
- AST 级或 hunk 级精细补丁修改
- 语义检索式知识库召回
- 托管式在线服务或 Web UI
- 以某一宿主平台为中心的深度平台绑定
- 以 plugin 为中心的产品形态

这些方向未来可以演进，但不属于当前 MVP 的边界。

## 7. 发布质量标准

当前版本可以视为达到开源 MVP 发布标准，当且仅当满足以下条件：

- 新用户按 README 可以成功跑通一次 mock 流程
- 新用户配置一个受支持的 provider 后，可以成功完成一次真实运行
- 失败时，用户能通过 CLI 输出和运行产物快速判断卡在哪
- 默认路径不依赖高风险自动执行能力
- 核心流程有自动化测试覆盖
- 文档明确区分“稳定主路径”和“进阶能力”

## 8. 当前阶段的工作优先级

结合当前实现情况，MVP 后续工作优先级应当是：

1. 打磨默认 CLI 体验和文档路径
2. 提升真实 provider 运行时的稳定性与报错清晰度
3. 明确支持矩阵和推荐使用路径
4. 保留执行闭环等高级能力，但降低其在默认路径中的权重

## 9. 总结

Maestro Flow 当前 MVP 的定位是：

`一个本地运行、面向开源发布、强调可审查与可追溯的多 Agent 交付 CLI`

它当前最应该对外传达的价值是：

- 结构化的多 Agent 编排
- 可复盘的运行产物
- 策略门禁与 CI 集成
- 人工控制的最终交付决策

当前最合适的发布策略是：

`把默认路径做简单、做稳定、做可信，同时把更强的自动化能力保留为可选进阶层。`
