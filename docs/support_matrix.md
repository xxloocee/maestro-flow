# Maestro Flow 支持矩阵与支持边界

## 1. 文档目的

本文档用于说明 Maestro Flow 当前版本对外发布时的支持范围，包括：

- 支持哪些宿主环境
- 支持哪些模型 provider
- 哪些能力属于稳定主路径
- 哪些能力属于高级能力或实验能力

本文档的目标不是列出“代码里已经出现的全部能力”，而是明确：

`哪些能力可以对外承诺，哪些能力暂时只适合谨慎使用`

## 2. 支持原则

当前版本遵循以下支持原则：

1. 产品本体是独立 CLI 工具  
   任何宿主环境支持，本质上都是对 CLI 的适配，不改变产品核心。

2. 默认优先支持“可审查的多 Agent 流程”  
   优先保障 requirement/spec -> 编排 -> 产物 -> review/CI 这一主路径。

3. 自动执行类能力不作为默认支持承诺  
   自动改代码、自动执行命令、隔离回写等能力已实现，但不作为第一次上手的默认能力承诺。

4. 宿主支持分层定义  
   区分“已完成适配”“推荐使用”“仍在增强”的不同层级，而不是简单写成支持/不支持。

## 3. 宿主环境支持矩阵

### 3.1 总览

| 宿主环境 | 当前状态 | 支持形式 | 说明 |
| --- | --- | --- | --- |
| `CLI` | 稳定 | 原生 | 产品本体，最优先支持 |
| `OpenCode` | 推荐 | slash commands | 已有实战项目验证，当前最推荐 |
| `Claude` | 可用 | slash commands | 已提供模板，适合作为通用接入 |
| `Cursor` | 可用 | slash commands | 已提供模板，适合作为通用接入 |
| `Antigravity` | 可用 | slash commands | 已提供模板，能力边界与 slash commands 一致 |
| `Codex` | 可用且推荐尝试 | skills | CLI、桌面端、IDE 扩展共享同一套 Codex skill 配置 |

### 3.2 分项说明

#### CLI

CLI 是产品本体，也是唯一必须稳定支持的入口。

当前对外发布的所有核心能力说明，都应默认以 CLI 为准。

#### OpenCode

当前已提供：

- `maestro-run`
- `maestro-spec`

OpenCode 已完成真实项目场景验证，当前版本建议作为最优先推荐的宿主环境。

#### Claude / Cursor / Antigravity

当前已提供 slash command 模板安装能力。

这类宿主的定位是：

- 能调用 CLI 工作流
- 能复用 requirement/spec 驱动能力
- 适合作为轻量接入层

当前不承诺宿主内更深层的自动化集成体验。

#### Codex

当前已提供的能力包括：

- `codex`：通过 skills 安装接入
  - `maestro-spec`
  - `maestro-run`
- 宿主内置 slash commands 可继续使用，但不作为当前仓库的自定义分发形式

当前限制包括：

- 尚未覆盖高级执行能力入口
- 宿主内体验仍主要围绕 CLI 映射，而不是更深的宿主能力整合
- 尚未形成和 CLI 同等完整的宿主体验

因此当前对 Codex 的官方定位应当是：

`可用且推荐尝试，但仍在增强中`

不建议在公开表述中把 Codex 描述为“完整深度支持”或“与 CLI 完全等价”。

## 4. Provider 支持矩阵

### 4.1 当前内置 provider

当前代码内置以下 provider 配置：

- `openai`
- `openrouter`
- `deepseek`
- `moonshot`
- `qwen`
- `siliconflow`
- `volcengine`
- `custom`

### 4.2 支持分层建议

对外发布时，建议将 provider 支持分为两层：

| Provider 类型 | 当前状态 | 说明 |
| --- | --- | --- |
| `openai` | 推荐优先验证 | 官方默认路径，最适合作为文档示例 |
| `openrouter` | 推荐优先验证 | OpenAI-compatible，适合多模型接入 |
| `custom` | 可用 | 适用于用户自定义 OpenAI-compatible 端点 |
| `deepseek` / `moonshot` / `qwen` / `siliconflow` / `volcengine` | 已内置支持 | 兼容路径已接入，但仍需持续实战验证 |

这里的“已内置支持”含义是：

- 代码层面已提供 provider profile
- CLI 可以通过环境变量切换
- 走的是 OpenAI-compatible 调用路径

这里不自动等同于“已完成充分实战验证”。

## 5. 能力支持矩阵

### 5.1 稳定主路径能力

以下能力属于当前对外可以明确承诺的稳定主路径：

| 能力 | 当前状态 | 说明 |
| --- | --- | --- |
| requirement 驱动运行 | 稳定 | `run --requirement` |
| spec 驱动运行 | 稳定 | `spec init` / `spec run` |
| 多 Agent DAG 编排 | 稳定 | `pm -> architect -> dev -> tester/debugger -> reviewer` |
| 阶段输出落盘 | 稳定 | `stage_<name>.json`、`summary.md`、`run_state.json` |
| 重试 / 状态机 / 错误码 | 稳定 | P0/P1 核心能力 |
| 知识库注入 | 稳定 | P4 |
| 提示词版本化 | 稳定 | P4 |
| 策略门禁 | 稳定 | P4/P5 |
| CI 评估与 PR 评论 | 稳定 | 当前 CLI 已接入 |
| 回滚机制 | 稳定 | P3 |

### 5.2 高级但非默认能力

以下能力已经实现，但不作为默认主路径承诺：

| 能力 | 当前状态 | 说明 |
| --- | --- | --- |
| 自动应用文件改动 | 高级能力 | 执行闭环的一部分 |
| 自动执行验证命令 | 高级能力 | 需要用户理解命令策略边界 |
| Debugger 自动修复迭代 | 高级能力 | 适合熟悉流程后使用 |
| 隔离副本执行 | 高级能力 | 推荐优先于直接原地执行 |
| sync-back 自动回写 | 高级能力 | 有冲突策略与回写边界 |
| 人工决策式回写冲突处理 | 高级能力 | 面向复杂工作区场景 |

### 5.3 当前不作为稳定承诺的部分

以下方向当前不应作为稳定承诺：

- 宿主平台内的深度自动化体验一致性
- AST 级或 hunk 级精细改写
- 更复杂的知识检索与召回
- 完全自治的软件交付
- 无人工审核的自动合并

## 6. 推荐对外表述

对外发布时，建议采用下面这类表述：

### 6.1 推荐表述

- Maestro Flow 是一个独立的开源 CLI 工具。
- 当前版本优先支持可审查、可追溯的多 Agent 软件交付流程。
- OpenCode 是当前最推荐的宿主集成路径。
- Claude、Cursor、Antigravity 提供轻量命令模板接入。
- Codex 已提供默认主路径 skills 接入，当前已可用并推荐尝试，但支持深度仍在增强。
- 自动执行闭环与回写能力已实现，当前定位为进阶能力。

### 6.2 不推荐表述

- 支持所有宿主并提供一致体验
- 已完整深度支持 Codex
- 默认推荐自动改代码并自动写回仓库
- 任意 OpenAI-compatible provider 都已充分验证

## 7. 后续工作指引

基于当前支持矩阵，后续工作建议按以下顺序推进：

1. 继续巩固 CLI 本体和默认主路径
2. 把 OpenCode 经验固化成回归样本
3. 补 Codex 适配深度，完善 skills 套件与宿主验证
4. 持续补充 provider 实战验证记录
5. 在支持边界足够清楚之后，再扩大高级能力的默认曝光度

## 8. 总结

当前版本最准确的对外定位是：

`CLI 本体稳定，OpenCode 推荐，Claude/Cursor/Antigravity 可用，Codex 已可用且推荐尝试，但仍需继续增强。`

能力层面最准确的定位是：

`默认主路径稳定，高级执行能力可用，但不应抢占默认产品叙事。`
